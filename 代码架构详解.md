# 串行摩擦力辨识程序架构说明

## 1. 目标与边界

本文档只描述“按电机顺序逐个辨识”的主流程，不再保留旧版的并行采集、并行辨识、全轴联合控制、仿真接入和刚体逆动力学链路。系统面对的是 7 个独立电机，而不是一个需要统一建模的 7 轴机械臂。

这份架构说明以如下约束为准：

1. 数据流严格按 `1 -> 2 -> 3 -> 4 -> 5 -> 6 -> 7 -> 1 -> ...` 的顺序循环。
2. 每一轮只控制并辨识一个目标电机。
3. 控制和记录都必须校验 `motor_id`，确保当前轮次只使用当前目标电机的反馈。
4. 辨识直接基于反馈值，不做反馈新鲜度判断，不做并行快照拼接，不接入仿真。

旧文档中出现的以下能力都不再属于必选项：`collect` 并行模式、全轴同步初始化、反馈新鲜度超时、7 维联合力矩向量、位置/速度/力矩限幅、软限位、MuJoCo、刚体逆动力学、末端姿态计算。

## 2. 系统主线

最小可复现链路可以概括为：

```text
run.sh -> 读取配置 -> 按 1~7 顺序选择目标电机 ->
生成当前电机激励 -> 读取串口反馈 -> 校验帧合法性与 motor_id ->
只在 id 匹配时更新当前电机控制并记录数据 ->
单轮结束后离线拟合当前电机摩擦参数 ->
保存单轮结果 -> 进入下一个电机 ->
全部轮次完成后聚合 summary
```

如果存在多组实验，则顺序必须是：

```text
group 1: 1 -> 2 -> 3 -> 4 -> 5 -> 6 -> 7
group 2: 1 -> 2 -> 3 -> 4 -> 5 -> 6 -> 7
group 3: 1 -> 2 -> 3 -> 4 -> 5 -> 6 -> 7
...
```

即使只启用部分电机，也必须按升序执行，例如只启用 `1,3,4` 时，顺序应为 `1 -> 3 -> 4 -> 1 -> 3 -> 4`。

## 3. 启动契约

程序仍建议保留两层入口：

1. 根目录 `run.sh`，只负责启动包装。
2. Python 主入口，负责模式解析、任务编排和结果落盘。

在这份新规范里，核心主流程只有 `sequential`。如果为了兼容旧脚本保留 `default`，则 `default` 应直接等价于 `sequential`。旧文档中的 `collect`、`compensate`、`compare` 不再是本规范的必需能力；即使暂时保留脚本入口，也不应再把它们写成架构主线。

Python 入口建议至少接受以下参数：

| 参数 | 说明 |
|---|---|
| `--config` | 配置文件路径 |
| `--mode` | 运行模式，默认 `sequential` |
| `--output` | 输出目录覆盖 |
| `--motors` | 目标电机集合，例如 `all` 或 `1,3,4` |
| `--groups` | 组数覆盖 |

其中 `--motors` 的解释必须是 1 基 `motor_id`，不是旧文档里的 0 基关节索引。

## 4. 配置契约

配置文件应围绕“单电机串行辨识”来组织，而不是围绕机械臂整机模型来组织。建议至少包含以下语义块：

| 配置块 | 必需内容 |
|---|---|
| `motors` | 电机 ID 列表、可选名称、默认启用集合 |
| `serial` | 串口设备、波特率、读超时、缓冲策略 |
| `excitation` | 单电机激励时长、幅值、频率段、启动保持时间、结束保持时间 |
| `control` | 当前电机控制所需的基础参数，例如 `kp`、`kd`、输出缩放 |
| `identification` | 平滑参数、正则化强度、Huber 阈值、最大迭代次数、最小样本数 |
| `output` | 结果根目录、文件命名规则、summary 文件名 |

以下内容不再要求出现在配置中：

1. URDF 路径。
2. MuJoCo 时间步长。
3. 关节上下限、软限位区域、边界余量。
4. 力矩上限、速度上限。
5. 反馈新鲜度超时因子。
6. 末端位姿、TCP、body 名称等整机建模参数。

如果需要数值稳定性，仍然可以保留一个很小的“零速去噪阈值”。但它只服务于拟合稳定，不应被写成“电机速度限制”。

## 5. 外部依赖与运行环境

本流程只需要最基础的数值计算和串口通信能力：

1. `numpy`，用于数组处理。
2. `scipy`，用于平滑和最小二乘拟合。
3. `pyserial`，用于串口收发。
4. 标准库中的 `json`、`pathlib`、`dataclasses`、`time` 等。

可视化、实时面板和仿真都不是主流程依赖。即使存在可视化模块，也必须是可选的，不能阻断辨识主流程。

## 6. 模块职责划分

建议至少保留以下八类职责：

1. 入口层：解析参数，决定本次要跑哪些电机、多少组。
2. 配置层：读取 YAML，产出结构化配置对象。
3. 激励层：生成单电机参考轨迹和阶段标签。
4. 串口协议层：解析上行帧、封装下行命令、处理重同步。
5. 控制层：仅基于当前目标电机的反馈更新当前轮次命令。
6. 记录层：把当前轮次的原始反馈、命令和元数据写入内存日志并落盘。
7. 辨识层：对单电机样本做平滑、筛样、拟合。
8. 结果层：保存单轮产物、聚合 summary、生成可读报告。

这里最重要的边界是：协议层负责“帧是否合法”，调度层负责“当前该控制哪个电机”，控制层负责“只用匹配 ID 的反馈做当前电机控制”，辨识层负责“只处理当前电机这一轮的样本”。

## 7. 串口上行协议

上行反馈仍然是单电机状态帧。推荐继续使用下面这个帧格式：

```text
<BBBffff
```

字段含义如下：

| 字段 | 类型 | 字节数 | 说明 |
|---|---:|---:|---|
| `head` | `uint8` | 1 | 固定帧头 `0xA5` |
| `motor_id` | `uint8` | 1 | 电机编号，范围 `1~7` |
| `state` | `uint8` | 1 | 电机状态字节 |
| `position` | `float32` | 4 | 位置反馈 |
| `velocity` | `float32` | 4 | 速度反馈 |
| `torque` | `float32` | 4 | 力矩反馈 |
| `mos_temperature` | `float32` | 4 | MOS 温度 |

总长度为 19 字节。由于协议里的 `motor_id` 是 1 基，因此业务层应直接使用 `motor_id` 本身，不需要再把它改写成“0 基关节索引”再回推。

如果系统内部仍想保留 `coil_temperature` 字段，也可以继续用 `NaN` 占位，但它不再是主流程必需字段。

## 8. 串口解析与顺序校验

上行解析不能按固定 19 字节盲切，必须做帧合法性检查。至少需要校验：

1. 帧头是否正确。
2. `motor_id` 是否位于 `1~7`。
3. `position / velocity / torque / mos_temperature` 是否都是有限值。

在此基础上，还应增加“顺序一致性检查”。因为当前数据流被明确规定为 `1 -> 2 -> 3 -> 4 -> 5 -> 6 -> 7` 循环，所以解析器应维护一个 `expected_motor_id`：

```text
1 -> 2 -> 3 -> 4 -> 5 -> 6 -> 7 -> 1 -> ...
```

如果收到的合法帧 `motor_id` 与 `expected_motor_id` 不一致，应记为顺序异常并触发重同步。这里的检查目的是保证协议顺序正确，不是为了做“反馈新鲜度判断”。

需要特别区分三种判断：

1. 帧是否合法。
2. 帧顺序是否符合 `1~7` 循环。
3. 这帧是不是当前目标电机的数据。

其中前两项属于通信校验，第三项属于控制校验。当前轮次只有在第 3 项成立时，才允许使用该帧做控制和记录。

## 9. 下行业务语义

这份规范不再把下行命令定义成“7 维联合力矩向量”。业务语义必须改成“当前轮次只对当前目标电机发送控制命令”。

如果现有设备协议已经支持单电机命令帧，那么控制层应直接发送当前目标电机的命令值。如果底层固件暂时仍要求某种固定长度帧，那么也必须由协议适配层负责兼容，不能把业务层重新写回“7 轴整机控制”的思路。

这里的关键不是下行字节数，而是业务含义：

1. 当前轮次只有一个 `target_motor_id`。
2. 当前命令只服务于这个 `target_motor_id`。
3. 非目标电机不参与本轮辨识，也不参与本轮控制决策。

由于电机被视为独立对象，本规范不要求默认的速度限幅或力矩钳位逻辑。若后续需要增加设备保护，那也应视为单独的安全策略，而不是这份架构的基础前提。

## 10. 模式语义

主流程只保留 `sequential`：

```python
for group_index in groups:
    for target_motor_id in enabled_motor_ids_in_ascending_order:
        run_single_motor_round(target_motor_id)
```

单轮的含义非常明确：

1. 选定一个 `target_motor_id`。
2. 对这个电机生成激励。
3. 只使用 `motor_id == target_motor_id` 的反馈做控制与记录。
4. 单轮结束后只对这个电机做一次离线辨识。
5. 保存该电机本轮的 `capture` 和 `identification`。

旧文档中的并行模式、并行采样、并行快照和全轴联合拟合逻辑全部删除，不再进入这份架构说明。

## 11. 单电机激励轨迹

参考轨迹只需要描述当前目标电机，不需要再维护 `N x 7` 的整机矩阵。建议至少包含：

| 字段 | 说明 |
|---|---|
| `time[N]` | 时间轴 |
| `position_cmd[N]` | 位置命令 |
| `velocity_cmd[N]` | 速度命令 |
| `phase_name[N]` | 阶段标签 |

如果控制器确实需要，也可以保留 `acceleration_cmd[N]`，但它不再是为了刚体逆动力学服务，而只是当前控制器的可选辅助量。

激励阶段建议至少包括：

1. 启动保持。
2. 正向运动段。
3. 反向运动段。
4. 多速度段或换向段。
5. 结束保持。

只要能覆盖正反向、低速和中高速样本即可，不要求照搬旧文档里针对机械臂整机写的激励逻辑。

## 12. 在线控制逻辑

每轮开始后，系统持续读取串口字节流，并按帧解析反馈。控制逻辑必须遵守下面这个约束：

1. 任何非法帧直接丢弃并继续寻找下一帧头。
2. 任何合法但 `motor_id != target_motor_id` 的帧，都不能作为当前轮次控制依据。
3. 只有 `motor_id == target_motor_id` 的帧，才能驱动当前电机命令更新和样本记录。

也就是说，本轮控制循环的关键条件不是“反馈够不够新鲜”，而是“当前拿到的是不是当前电机的数据”。

一个最小可复现的单轮控制循环可以表示为：

```python
while not round_finished:
    frame = read_next_valid_frame()
    check_sequence(frame.motor_id)
    if frame.motor_id != target_motor_id:
        continue

    feedback = frame_to_feedback(frame)
    command = controller.update(reference_at_now, feedback)
    send_command(target_motor_id, command)
    logger.append(now, target_motor_id, reference_at_now, feedback, command)
```

本流程不再要求：

1. 等待全部 7 个电机都初始化。
2. 等待“全 active motors 新鲜快照”。
3. 统计“自上轮以来是否所有电机都收到新反馈”。

目标电机拿到第一帧合法且 ID 匹配的反馈后，就可以进入正常控制和记录。

## 13. 原始采样数据契约

由于每一轮只针对一个电机，采样对象应改成“单电机标量序列”，而不是旧文档中的 `N x 7` 矩阵。建议至少包含：

| 字段 | 说明 |
|---|---|
| `time[N]` | 采样时间 |
| `motor_id[N]` | 当前轮次的目标电机 ID，通常整列相同 |
| `position[N]` | 位置反馈 |
| `velocity[N]` | 速度反馈 |
| `torque_feedback[N]` | 力矩反馈 |
| `command[N]` | 下行命令值 |
| `position_cmd[N]` | 位置命令 |
| `velocity_cmd[N]` | 速度命令 |
| `phase_name[N]` | 阶段标签 |
| `state[N]` | 状态字节 |
| `mos_temperature[N]` | MOS 温度 |
| `id_match_ok[N]` | 本条样本是否来自当前目标电机，正常情况下应全为真 |
| `metadata` | 轮次元数据 |

`metadata` 至少应包含：

1. `group_index`
2. `round_index`
3. `target_motor_id`
4. `enabled_motor_ids`
5. `excitation_config`
6. `start_time`
7. `stop_reason`
8. `sequence_error_count`

旧文档中的这些字段可以直接移除：`range_ratio`、`limit_margin_remaining`、`joint_refresh_mask`、`ee_pos`、`ee_quat`、`tau_track_ff`、`tau_track_fb`、`tau_residual`、全轴温度矩阵、全轴 UART 面板字段。

## 14. 离线辨识前处理

单轮采样结束后，离线处理应尽量保持简单，核心目标是把“当前电机的反馈序列”整理成可拟合样本。

建议流程如下：

1. 检查样本数是否达到最低要求。
2. 对速度做平滑，优先使用 Savitzky-Golay；样本过少时退回简单方法。
3. 以反馈力矩作为辨识目标量。
4. 根据数值有效性和运动充分性构造样本掩码。

这里最关键的变更是：不再做刚体项扣除，不再计算 `tau_measured - tau_rigid`，也不再依赖任何 frictionless 模型。当前电机的辨识目标应直接来自反馈值。

即：

```text
tau_target = torque_feedback
```

如果后续还想保留一个加速度序列，也只能作为辅助分析字段，而不是仿真或逆动力学的输入前提。

## 15. 样本有效性判定

新的筛样逻辑应以“当前电机、当前反馈、当前轮次”为中心，而不是以整机状态为中心。建议至少保留以下条件：

1. `position / velocity / torque_feedback` 为有限值。
2. `motor_id == target_motor_id`。
3. 当前轮次有足够的正反向运动样本。
4. 当前轮次总样本数不低于最低要求。

可以额外保留两类数值稳定措施：

1. 很小的零速去噪阈值，用来减少换向点噪声。
2. 最小运动跨度约束，用来排除“几乎没动”的无效轮次。

但以下旧条件应明确删除：

1. 反馈新鲜度判定。
2. 全轴初始化判定。
3. 空间工作区判定。
4. 基于力矩上限的异常裁剪。
5. 基于机械臂边界的软限位筛样。

## 16. 辨识输入契约

真正送入拟合器的输入应简化为当前电机的一组标量序列：

| 字段 | 说明 |
|---|---|
| `velocity[N]` | 平滑后的速度 |
| `torque[N]` | 反馈力矩目标值 |
| `sample_mask[N]` | 样本是否有效 |
| `motor_id` | 当前电机 ID |
| `metadata` | 样本数、速度统计、阶段统计等 |

如果实现上为了复用矩阵代码仍想保留二维形状，也应视为 `N x 1`，而不是重新回到多电机联合输入。

## 17. 摩擦模型与拟合方法

摩擦模型仍可以保持为：

```text
tau_friction = coulomb * tanh(v / velocity_scale) + viscous * v + offset
```

这里的拟合对象就是当前电机本轮样本，不存在多电机耦合，也不存在并行联合建模。

推荐保留以下拟合特性：

1. 带正则化的最小二乘。
2. Huber 型鲁棒重加权。
3. `velocity_scale` 的候选搜索。
4. 正反向样本不均衡时的权重修正。

只要始终满足“单电机单轮独立求解”，具体实现可以灵活调整。

## 18. 训练集与验证集

如果需要模型选择，仍然可以为当前电机当前轮次划分训练集和验证集。推荐做法是：

1. 先根据 `sample_mask` 取出有效样本。
2. 再按固定步长抽取少量验证样本。
3. 启动初期样本可不进入验证集，以避开瞬态。

如果某轮没有足够验证样本，则可以退回只看训练误差。这个机制只在当前单电机样本内部进行，不再与其他电机共享时间轴或验证策略。

## 19. 单轮结果契约

每个电机每轮结束后，应至少产出两类文件：

1. `capture`：原始采样数据。
2. `identification`：离线拟合结果。

`identification` 至少建议包含：

| 字段 | 说明 |
|---|---|
| `motor_id` | 当前电机 ID |
| `coulomb` | 库仑摩擦参数 |
| `viscous` | 粘性摩擦参数 |
| `offset` | 偏置项 |
| `velocity_scale` | 平滑符号尺度 |
| `torque_pred[N]` | 模型预测力矩 |
| `torque_target[N]` | 拟合目标力矩 |
| `train_mask[N]` | 训练样本掩码 |
| `valid_mask[N]` | 验证样本掩码 |
| `train_rmse` | 训练误差 |
| `valid_rmse` | 验证误差 |
| `train_r2` | 训练 R² |
| `valid_r2` | 验证 R² |

由于每次只拟合一个电机，这里不再需要“第二维是不定长 `J_fit`”的矩阵定义。

## 20. Summary 聚合规则

当所有组、所有电机都跑完后，应生成一个按 `motor_id` 聚合的 summary。建议 summary 至少包含长度为 7 的数组：

1. `motor_ids`
2. `identified_mask`
3. `coulomb`
4. `viscous`
5. `offset`
6. `velocity_scale`
7. `validation_rmse`
8. `validation_r2`
9. `sample_count`
10. `valid_sample_ratio`

聚合规则建议如下：

1. 对同一电机的多轮有效参数取有限均值。
2. 对同一电机的误差指标取有限均值。
3. 对同一电机的参数一致性保留标准差。
4. 对同一电机保留每轮历史值，便于回看。

也就是说，summary 是“按电机汇总单轮结果”，不是“按并行批次汇总全轴快照”。

## 21. 输出目录与文件契约

结果目录建议采用“每次运行独立归档 + 根目录镜像 summary”的结构：

```text
results/
  hardware_identification_summary.npz
  hardware_identification_summary.csv
  hardware_identification_report.md
  runs/
    20260418_120000_sequential/
      group_01/
        motor_01/
          capture.npz
          identification.npz
        motor_02/
          capture.npz
          identification.npz
        ...
      group_02/
        motor_01/
        ...
      summary/
        identification_summary.npz
        identification_summary.csv
        identification_report.md
      run_manifest.json
```

`run_manifest.json` 至少应记录：

1. `run_label`
2. `mode`
3. `start_time`
4. `end_time`
5. `group_count`
6. `motor_order`
7. `capture_files`
8. `identification_files`
9. `summary_files`

其中 `motor_order` 必须能明确反映本次运行采用的是哪一种顺序，例如 `[1,2,3,4,5,6,7]`。

## 22. 明确删除的旧逻辑

为了避免后续再次把架构写偏，下面这些逻辑应明确视为“已删除”：

1. 并行采集。
2. 并行辨识。
3. 全轴状态拼接。
4. 反馈新鲜度判定。
5. 全轴完整快照等待。
6. 7 维联合力矩向量作为业务前提。
7. 位置窗口、软限位、边界余量。
8. 速度上限、力矩上限作为基础控制逻辑。
9. MuJoCo、URDF、正运动学、逆动力学。
10. 基于刚体残差的辨识输入。

只要把这些旧前提删干净，程序的真实主线就会重新收敛到“单电机、按 ID、按顺序、基于反馈值逐个辨识”。

## 23. 一句话摘要

这套程序现在应被理解为：一个面向 7 个独立电机、按 `1~7` 固定顺序循环、在每一轮只使用当前 `motor_id` 匹配反馈做控制与记录、并直接基于该电机反馈值完成摩擦参数拟合的串行辨识系统。
