# OpenArm 摩擦力辨识程序复现规范

## 1. 目标与边界

本规范用于复现当前仓库中的真实程序行为，而不是复现最早的规划版本。系统的目标是对 7 轴机械臂执行真机摩擦力辨识，并将辨识结果反向用于摩擦补偿验证。当前实现已经收敛为真机专用链路，MuJoCo 的职责不是做仿真主流程，而是提供刚体逆动力学和正运动学支撑。程序必须支持四种模式：`sequential`、`collect`、`compensate`、`compare`。其中 `sequential` 是默认主流程，`collect` 是保留的并行兼容模式，`compensate` 读取已有参数做纯摩擦补偿验证，`compare` 用于比较历史辨识结果。

如果目标是“让另一个 AI 重新写一套程序，并保证行为与当前仓库一致”，那么需要优先保证下面五件事一致：一是启动方式和模式语义一致；二是串口上下行协议一致；三是采集时序与控制逻辑一致；四是辨识输入构造和拟合模型一致；五是输出文件结构与关键字段一致。仓库中没有 `run.h`，根目录真实入口是 `run.sh`，所以复现入口也应以 `run.sh + python -m` 的双层结构来实现。

## 2. 复现后的系统应长什么样

复现后的系统本质上是一个“硬件在线采集 + 离线摩擦拟合 + 结果归档 + 参数回放验证”的流水线。它不是纯控制程序，也不是纯辨识程序，而是一个围绕实验任务组织的完整闭环系统。上层负责接受模式、配置和输出目录等外部输入；中层负责生成参考轨迹、读取串口反馈、计算下发力矩、实时记录关键状态；下层负责在采样结束后做速度平滑、加速度重建、刚体项扣除、样本筛选和摩擦拟合；最后一层负责把单轮结果和最终汇总写成 NPZ、CSV、Markdown 和 JSON，并在需要时再读取 summary 生成摩擦补偿力矩。

这个系统的最小可复现主线可以概括为：

```text
启动脚本 -> 读取配置 -> 选择模式 -> 生成参考轨迹 -> 建立串口 -> 读取反馈 -> 计算控制力矩 ->
记录采样数据 -> 结束后构建辨识输入 -> 拟合摩擦参数 -> 写入归档和 summary ->
在补偿模式中反向读取 summary 生成摩擦补偿力矩
```

## 3. 启动契约

复现后的程序必须保留两层启动入口。第一层是根目录 `run.sh`，面向人工操作，提供交互菜单、上次运行记录、配置切换和快捷启动。第二层是真正的 Python 程序入口，负责解析命令行参数并执行模式逻辑。`run.sh` 不承载业务核心，只做启动包装；业务核心全部放在 Python 入口和后续模块中。

`run.sh` 必须支持以下行为：无参数启动时显示交互菜单；`default` 表示按 YAML 中的默认模式运行；`sequential` 表示逐关节辨识；`collect` 表示 7 轴并行采集与并行辨识；`compensate` 表示补偿验证；`compare` 表示比较历史结果。脚本还需要支持把 `--output`、`--compare-all` 之类的额外参数原样透传给 Python 程序。配置层中的 `parallel` 在执行层要映射成 `collect`，也就是说 YAML 中写的是 `parallel`，实际运行时 CLI 模式名是 `collect`。这点必须保留，否则默认模式的行为会与现有仓库不一致。

Python 入口必须接受至少以下参数：配置文件路径、运行模式、输出目录覆盖、关节覆盖、对比数量、是否比较全部历史结果。关节覆盖格式应允许 `"all"` 或 `"0,2,4"` 这类字符串，并将其解释为 0 基关节索引集合。覆盖后的关节集必须替换配置中的 `active_joints`，并在后续采集、辨识和汇总中生效。

## 4. 配置契约

程序必须使用一个结构化 YAML 作为统一配置入口。为了保证可复现性，配置对象至少应包含以下语义块：机器人模型、辨识模式与激励参数、控制参数、安全参数、采样参数、辨识参数、串口参数、可视化参数、输出参数。配置加载阶段必须完成严格校验，而不是在运行中再被动发现错误。

机器人配置必须提供：URDF 路径、7 个关节名称、7 个关节上下限、7 个关节力矩上限、回零位 `home_qpos`、TCP 偏移和末端 body 名称。辨识配置必须提供：当前模式、激活关节索引、逐关节模式参数、激励轨迹参数和启动过渡参数。控制配置必须提供每个关节的 `kp`、`kd` 以及反馈缩放系数。安全配置必须提供关节边界裕量、是否启用力矩钳位和软限位区域。采样配置必须提供参考采样率、MuJoCo 时间步长和真机参考更新系数。辨识配置必须提供速度平滑尺度、正则化强度、最大迭代次数、Huber 阈值和最小有效速度阈值。串口配置必须提供设备名、波特率和反馈新鲜超时因子。输出配置必须定义结果根目录、采样文件前缀、辨识文件前缀、summary 文件名、report 文件名和兼容 JSON 文件名。

当前真实默认配置还有一个特别重要的事实：激励窗口默认是 `unbounded`。这意味着当前位置安全窗口不会像 `safe` 或 `hard` 那样对轨迹构造形成硬边界，控制中也不会触发常规的限位位置判定。这种行为必须被理解为“当前实现更偏向速度激励和实验可用性”，而不是“严格位置窗安全模式”。如果要完全复现当前行为，就必须允许 `unbounded` 模式使关节窗口变成 `(-inf, +inf)`，从而让范围比缺省为 `0.5`、边界余量缺省为 `NaN`、软限位衰减基本不参与。

## 5. 外部依赖与运行环境

程序运行至少依赖以下能力：一是 Python 数值计算环境，用于数组、插值、优化和文件保存；二是 `pyserial`，用于真机串口通信；三是 `mujoco`，用于根据 URDF 构建模型并执行刚体逆动力学与正运动学；四是 `scipy`，用于 Savitzky-Golay 滤波和带约束最小二乘；五是 `rerun`，用于实时仪表板，但它必须是可选的，可视化失败不能阻断主流程。复现实现也应维持这种“可视化可选、主流程不能依赖可视化”的原则。

## 6. 模块职责划分

为了让另一个 AI 容易重写，建议按照职责而不是按源码函数来组织模块。需要至少保留八类职责。第一类是入口与模式调度，负责解析外部输入并决定本次运行走哪条链路。第二类是配置与公共数据结构，负责把 YAML 变成强类型配置对象，并统一定义采样数据、辨识输入和辨识结果的结构。第三类是参考轨迹层，负责生成位置、速度、加速度命令以及阶段标签。第四类是控制层，负责结合参考轨迹和当前反馈生成力矩命令。第五类是真机接口层，负责串口上行解析、串口下行打包、全局状态缓存、控制周期驱动和原始数据记录。第六类是辨识准备层，负责速度滤波、加速度重建、刚体项扣除和样本筛选。第七类是辨识层，负责按关节独立拟合摩擦参数。第八类是结果层，负责保存每轮产物、聚合 summary、生成 Markdown/CSV/JSON，并在 compare 模式下扫描历史归档。

这八类职责必须保持清晰边界。启动层不做算法；轨迹层不直接访问串口；辨识层不关心目录结构；结果层不参与控制；真机接口层负责把“连续字节流和实时循环”转成“结构化实验数据”。只有这样，另一个 AI 才能在不依赖原函数名的情况下稳定复现整体行为。

## 7. 串口上行协议

上行指电机控制板向上位机反馈的单电机状态帧。当前程序假定反馈流不是“7 轴大帧”，而是“按单电机逐帧上送”的字节流。复现实现必须延续这个前提，因为后续的反馈新鲜度判断、全轴状态拼接和并行快照逻辑都建立在这个事实之上。

单帧反馈格式必须是：

```text
<BBBffff
```

字段顺序和含义如下：

| 字段 | 类型 | 字节数 | 说明 |
|---|---:|---:|---|
| `head` | `uint8` | 1 | 固定帧头 `0xA5` |
| `motor_id` | `uint8` | 1 | 电机编号，范围 1 到 7 |
| `state` | `uint8` | 1 | 电机状态字节 |
| `position` | `float32` | 4 | 关节位置反馈 |
| `velocity` | `float32` | 4 | 关节速度反馈 |
| `torque` | `float32` | 4 | 关节力矩反馈 |
| `mos_temperature` | `float32` | 4 | MOS 温度 |

总长度是 19 字节。程序内部采用 0 基关节索引，但协议中的 `motor_id` 是 1 基，因此必须执行 `joint_index = motor_id - 1` 的映射。复现时还必须保留一个实现细节：当前上行帧并不携带线圈温度，因此系统内部统一创建一个 `coil_temperature` 占位量，值为 `NaN`，以便存储格式和可视化结构保持不变。

上行解析不能简单按 19 字节切片后盲目解包，而必须做候选帧合法性验证。合法性要求至少包括：帧头正确、`motor_id` 位于 1 到 7、数值字段有限、如果缓存中已经有两个完整帧长度，则下一帧边界也应对齐到帧头。任何不满足这些条件的候选都应该被视为噪声字节流并继续滑窗查找下一帧头。这一策略是当前程序避免串口边界错位的重要基础。

## 8. 串口下行协议

下行指上位机向真机发送的 7 轴力矩命令帧。当前实现始终发送固定长度的 7 轴力矩向量，不存在“单关节命令帧”，即便在 sequential 模式下也发送完整 7 维向量，只是非目标关节的力矩由控制器逻辑自然给出或保持为零。

下行帧格式必须是：

```text
0xAA 0x55 + tau1..tau7(float32 little-endian) + xor + 0x55 0xAA
```

更结构化地写为：

| 字段 | 类型 | 字节数 | 说明 |
|---|---:|---:|---|
| `head` | 2 字节 | 2 | 固定头 `0xAA 0x55` |
| `tau1..tau7` | 7 个 `float32` | 28 | 7 个关节的目标力矩 |
| `checksum` | `uint8` | 1 | XOR 校验 |
| `tail` | 2 字节 | 2 | 固定尾 `0x55 0xAA` |

总长度是 33 字节。当前实现中的 XOR 计算覆盖“帧头 + 28 字节负载”，而不是只覆盖负载。这一点必须原样保留，否则设备端校验可能不通过。

下行力矩向量的业务语义随模式不同而不同。在 `sequential` 和 `collect` 模式中，下行力矩来自“刚体逆动力学前馈 + 误差反馈修正”再经过力矩钳位和软化处理。在 `compensate` 模式中，下行力矩只来自摩擦模型预测结果，不包含轨迹前馈，不包含反馈校正，也不混入额外的全前馈补偿。这是当前程序的关键约束，复现时必须保持。

## 9. 模式语义

`sequential` 是当前默认主流程。它的目标不是同时辨识所有关节，而是按关节顺序逐个采集和拟合。整个运行由“组”和“关节轮次”两层构成：系统遍历每一组，在组内再遍历所有 active joints，每一轮只让一个目标关节执行单关节激励，其余关节保持当前控制体系要求的状态。每轮结束后做单轮辨识并保存，所有轮次结束后做统一 summary。当前实现还支持 sequential 的断点续跑，即如果归档中存在同配置、未完成且未标记结束的 sequential 运行，则恢复未完成轮次并继续。

`collect` 是保留的并行采集模式。它会为所有 active joints 同时生成全轴参考轨迹，同时采样并同时做后处理，但辨识阶段仍然是逐关节独立拟合，不做关节耦合联合建模。`collect` 最复杂的部分不在拟合，而在控制推进条件：程序必须判断当前 active joints 是否都已经初始化、反馈是否足够新鲜，以及本轮 emit 之后是否确实收到过新的反馈。只有满足这些条件，系统才允许本轮样本进入控制和记录。

`compensate` 的目标不是重新辨识，而是读取已有 summary 中的摩擦参数，根据当前反馈速度计算摩擦补偿力矩并直接下发，记录补偿期的状态与力矩数据。程序必须允许在没有历史 summary 时继续运行，此时补偿力矩向量全零，并给出提示。

`compare` 的目标是扫描历史归档、读取 summary、生成跨运行对比报告。当前真实实现只比较 `collect` 归档，而不是全量比较 `sequential` 归档。复现时如果目标是严格对齐现有行为，应保留这一限制。

## 10. 参考轨迹契约

程序中的参考轨迹是一个时间索引结构，必须至少包含五类信息：时间轴 `time[N]`、位置命令 `q_cmd[N,7]`、速度命令 `qd_cmd[N,7]`、加速度命令 `qdd_cmd[N,7]` 和阶段标签 `phase_name[N]`。无论是 sequential 还是 collect，都必须把轨迹统一表示成这个形状，以便控制层和记录层无需知道上层是单关节还是全轴模式。

当前激励轨迹不是简单正弦，而是分阶段组织。阶段至少包括启动对位、全范围扫动、换向抖动、多速度段扫动和末端保持。复现时不要求一字不差地沿用原始内部实现细节，但必须保留相同的行为目标：提供双向运动样本、换向附近样本、低速与中高速样本、可区分阶段的标签输出。当前实现中阶段标签采用字符串，以便直接写入记录和可视化。

在 `sequential` 模式下，只允许目标关节被视为 active，其余关节命令保持在 `home_qpos`。在 `collect` 模式下，所有 active joints 同时参与激励，并通过相位偏移降低完全同相带来的耦合和峰值风险。当前默认 `unbounded` 窗口下，轨迹构造更偏向速度主导，即先生成速度曲线，再积分得到位置，再通过梯度得到加速度；在 `safe` 或 `hard` 窗口下，轨迹则更偏向位置扫描，并受到边界约束。

此外，还必须保留启动过渡轨迹。系统在进入激励之前，需要从当前实际姿态平稳过渡到激励起始姿态。当前实现使用五次多项式轨迹完成这个过渡，并根据末端速度约束自动决定过渡时长。复现时至少要保证：如果当前姿态和目标姿态不同，则先做平滑过渡；如果二者足够接近，则可跳过启动轨迹。

## 11. 在线控制逻辑

在线控制逻辑是整个程序的实时核心。程序启动后应先建立串口，清空输入缓存，立即发送一个全零力矩帧作为安全初值。之后进入非阻塞循环：持续从串口缓冲中读取所有可用字节，解析出尽可能多的反馈帧，并把每一帧写入当前全局状态缓存。这些缓存至少包括位置、速度、实测力矩、MOS 温度、占位线圈温度以及每个关节最后一次更新时间。

控制推进条件根据模式不同而不同。在 `sequential` 模式下，程序只需要等待目标关节先收到第一帧反馈，一旦目标关节完成初始化，后续控制就按周期推进，不再要求每一轮都重新凑齐“全 active joints 完整新鲜快照”。在 `collect` 模式下，推进条件更严格：所有 active joints 必须都已初始化，所有 active joints 的反馈必须处于新鲜时间窗内，并且自上一次记录以来必须收到过新的 active joint 反馈。只有在满足这些条件后，当前控制周期才被认为是可用周期。

每个有效控制周期必须完成以下动作。首先确定当前已经过去的实验时间，并从参考轨迹中取出同一时间点的命令位置、命令速度、命令加速度以及阶段标签。然后由刚体逆动力学求得前馈力矩，再由当前位置和速度误差生成反馈修正，按配置缩放后叠加到前馈力矩上，形成采集模式下的力矩命令。在 `compensate` 模式下，则直接根据当前速度和已有摩擦参数计算摩擦补偿力矩，并将它作为唯一的下行命令。命令生成后必须经过力矩钳位，如果窗口模式允许，还应在接近边界时对外向力矩做软衰减。

当前实现还在每个周期在线估计一个粗略的加速度，即用当前速度和上周期速度做差分。利用这个在线加速度和当前姿态、速度，可以实时重算一个刚体逆动力学力矩，并构造在线残差力矩 `tau_residual = tau_measured - tau_rigid_live`。这个量用于在线监控和写入记录，但最终离线辨识仍会重新计算更干净的版本。

在每个有效周期结束时，程序必须把当前全套状态追加到内存日志中，包括时间、真实位置速度、命令位置速度加速度、实测力矩、下发力矩、前馈项、反馈项、补偿项、残差项、刷新掩码、旋转状态、范围比、边界余量、阶段标签、温度、通信指标和末端姿态。只有在这些日志累计完成后，本轮采样才算真正有了可离线处理的数据基础。

## 12. 在线状态派生与监控量

程序不仅记录“原始反馈”和“命令”，还必须实时派生若干状态量，因为这些量既用于调试，也用于后续筛样。第一类是旋转状态。规则很简单：当速度大于正阈值时记为 `+1`，当速度小于负阈值时记为 `-1`，落在阈值窗内则记为 `0`。这个状态用于判断正转、反转和近似静止。第二类是范围比。它表示当前位置在当前允许工作区间中的归一化位置，数值通常落在 0 到 1 之间；在 `unbounded` 窗口下，由于没有有限边界，这个值应退化为固定 `0.5`。第三类是边界余量，它表示当前位置距离上下边界较近一侧的剩余距离；在 `unbounded` 窗口下应为 `NaN`。第四类是通信健康度，包括 UART 周期频率、UART 周期时延和吞吐率。它们不参与辨识，但对于判断串口是否稳定非常关键。

复现实现还应保留阶段标签，因为阶段标签使得可视化和问题排查更容易。当前程序不仅把阶段标签写入每个采样点，还把它转成数值曲线用于 `rerun`，但这属于显示层，可选实现；核心要求是原始标签必须进入采样文件。

## 13. 原始采样数据契约

每一轮采样结束后，程序应生成一个统一的结构化采样对象，并最终保存为 `capture` 类 NPZ 文件。这个对象必须至少包含：时间 `time[N]`、真实位置 `q[N,7]`、真实速度 `qd[N,7]`、命令位置 `q_cmd[N,7]`、命令速度 `qd_cmd[N,7]`、命令加速度 `qdd_cmd[N,7]`、下发力矩 `tau_command[N,7]`、实测力矩 `tau_measured[N,7]`、前馈项 `tau_track_ff[N,7]`、反馈项 `tau_track_fb[N,7]`、摩擦补偿项 `tau_friction_comp[N,7]`、在线残差力矩 `tau_residual[N,7]`、刷新掩码 `joint_refresh_mask[N,7]`、旋转状态 `rotation_state[N,7]`、范围比 `range_ratio[N,7]`、边界余量 `limit_margin_remaining[N,7]`、批次或轮次号 `batch_index[N]`、阶段标签 `phase_name[N]`、MOS 温度 `mos_temperature[N,7]`、线圈温度 `coil_temperature[N,7]`、末端位置 `ee_pos[N,3]`、末端姿态 `ee_quat[N,4]`、UART 周期频率 `uart_cycle_hz[N]`、UART 时延 `uart_latency_ms[N]`、UART 吞吐率 `uart_transfer_kbps[N]` 和元数据 `metadata`。

元数据必须足以恢复本轮语义，至少包括：运行模式、批次或轮次索引、总轮次数、组索引、目标关节索引、最终拟合关节索引、激活关节索引、终止原因、实时有效样本比例。复现时不要求元数据字段名完全与原仓库一致，但必须表达同样的信息。

## 14. 离线辨识前处理

当前程序在采样结束后不会直接用在线记录的速度和残差做拟合，而是进行一次专门的离线辨识前处理。首先，若样本总数过少，则直接放弃辨识。然后确定本轮真正候选的拟合关节集合。这个集合的起点是配置中的 active joints，但在 `sequential` 中通常只包含一个目标关节，在后续运动充分性筛选后还可能继续缩小。

接下来必须对整段速度做平滑。当前实现优先采用 Savitzky-Golay 滤波；只有在样本过少、不足以满足窗口长度和多项式阶数要求时，才退回简单的数值梯度。平滑完成后，使用同一套平滑策略重建离线加速度。然后使用一个“去掉摩擦和阻尼项的刚体模型”对整段数据逐样本重算逆动力学，从而得到离线刚体项。当前残差的正式定义是：

```text
tau_residual = tau_measured - tau_rigid
```

这里的 `tau_rigid` 必须来自一个 frictionless 的刚体逆动力学模型，也就是说模型中各关节的 `frictionloss` 和 `damping` 需要被清零。只有这样，残差才可以被解释为“主要包含摩擦与未建模误差”的目标量。

## 15. 样本有效性判定

为了保证辨识结果稳定，复现实现必须保留多层筛样机制。第一层是数值有效性，要求位置、速度和残差力矩都是有限值。第二层是空间有效性，要求样本位于当前允许工作区间内；如果窗口模式是 `unbounded`，这一条件天然通过。第三层是运动有效性，要求速度绝对值不低于最小速度阈值，避免把零速附近的大量噪声样本直接送入拟合。第四层是残差合理性，要求残差力矩绝对值不超过关节力矩上限的一定倍数，当前实现是 1.5 倍。第五层是刷新有效性，要求当前关节在这个时刻被认为是“新鲜反馈”。

除了逐样本筛选，程序还要做逐关节运动充分性筛选。当前逻辑并不满足于“命令轨迹对这个关节发过运动指令”，而是进一步要求：该关节的实际位置跨度不能太小，实际速度 95 分位不能太小，并且这两个量不能远低于命令所要求的幅度。换句话说，如果命令上看该关节应该运动很多，但真实反馈只呈现出轻微编码器抖动，那么这个关节应被认定为“本轮没有形成有效可辨识运动”，从最终拟合集合中移除。这个设计是当前程序避免“关节没真正动起来却误导拟合器”的关键。

最后还必须有最小样本数约束。当前实现中，如果某关节的训练样本少于下限，则该关节本轮参数记为无效；如果整轮所有关节都无法保留足够样本，则整轮跳过辨识，只保留采样结果。

## 16. 辨识输入契约

经过离线前处理和筛样后，真正送入拟合器的不是原始采样对象，而是一组更紧凑的辨识输入。它至少包含三个核心量：速度矩阵 `velocity[N,J_fit]`、目标力矩矩阵 `torque[N,J_fit]` 和样本掩码 `sample_mask[N,J_fit]`。其中 `velocity` 来自平滑后的速度，`torque` 来自离线残差力矩，`sample_mask` 表示每个时刻对每个拟合关节该样本是否可用。除此之外，还需要关节名称列表和一些辅助元数据，例如保留样本数、每关节有效比、哪些关节在运动充分性筛选后幸存。

这个阶段非常重要，因为它把“原始实验记录”转换成“纯粹的辨识问题”。如果复现时不做这一步，而是把在线记录直接丢给拟合器，那么结果会与当前程序明显偏离。

## 17. 摩擦模型与拟合方法

当前程序对每个关节采用相同的摩擦模型：

```text
tau_friction = coulomb * tanh(v / velocity_scale) + viscous * v + offset
```

这意味着拟合特征由三列组成：平滑符号项、速度项和常数项。这里不使用硬 `sign(v)`，而使用 `tanh(v / velocity_scale)`，目的是让零速附近更平滑，避免数值优化在换向点附近变得极不稳定。拟合器不是做全轴联合建模，而是对每个拟合关节独立求解参数，因此即便是在 `collect` 模式中，数据是同时采的，参数仍然是逐关节独立估计。

拟合方法必须包含四个要点。第一，要用带正则化的最小二乘，而不是纯裸最小二乘。第二，要加入平衡性权重，降低正向样本和反向样本严重不均衡时的偏差。第三，要加入 Huber 型鲁棒重加权迭代，以减少异常值对参数的破坏。第四，要对 `velocity_scale` 做候选搜索，而不是固定死用一个值。当前实现会先根据速度分布自动构造一组候选尺度，再在这些尺度上尝试不同的参数模型，并用验证误差选出更优解。

为了保证结果稳定，参数还要带约束。当前实现中，库仑项和粘性项默认要求非负，偏置项允许在一个很小的正负范围内浮动。如果复现目标是严格对齐当前程序，就应保留这一约束思路。

## 18. 训练集与验证集策略

当前程序不是把全部有效样本都用于训练，而是自动划出一个周期性验证集。策略可以概括为：在有效样本序列中按固定步长抽点作为验证样本，但最开始的一段样本不参与验证，以避开启动瞬态。训练集是“有效样本减去验证样本”。如果复现目标是与当前行为尽量接近，那么验证集也必须是按关节独立构造，而不是一刀切地对整段时间轴统一抽样。

程序在不同候选参数组合之间做选择时，优先看验证误差；如果某一轮某关节没有有效验证样本，则退回使用训练误差进行比较。最终输出的误差指标至少包括每关节的训练 RMSE、验证 RMSE、训练 R² 和验证 R²。

## 19. 单轮辨识结果契约

每轮拟合完成后，应生成一个结构化辨识结果对象，并保存为 `identification` 类 NPZ 文件。这个结果至少要包含：本轮参与拟合的关节索引、每个关节的 `coulomb`、`viscous`、`offset`、`velocity_scale`、模型预测力矩、用于拟合的目标力矩、训练集掩码、验证集掩码、训练 RMSE、验证 RMSE、训练 R² 和验证 R²。由于拟合是逐关节独立进行的，这些矩阵的第二维是本轮幸存的拟合关节数 `J_fit`，不一定等于 7。

辨识结果必须与采样结果分开保存。这样做的目的不是重复写盘，而是明确地区分“原始实验记录”和“离线建模产物”。一旦后续需要重做拟合，就可以只重用 capture 文件，而不需要重新上机采样。

## 20. Summary 聚合规则

当前程序在所有轮次结束后，会生成一个最终 summary。这个 summary 的本质是“按关节聚合所有单轮结果”。聚合时必须至少执行以下规则。对于每个关节，汇总后的 `coulomb`、`viscous`、`offset` 取所有有效单轮结果的有限均值；汇总后的 `validation_rmse` 和 `validation_r2` 也取有限均值；一致性标准差取多轮参数的有限标准差；该关节的 `sample_count` 取多轮中保留下来的最大样本数；`valid_sample_ratio` 取多轮平均有效比例。最终 `identified_mask` 只在当前 active joints 范围内生效，并且要求该关节汇总参数是有限值。

summary 中还必须保留每轮历史矩阵，例如 `batch_coulomb`、`batch_viscous`、`batch_offset`、`batch_validation_rmse`、`batch_validation_r2`、`batch_sample_count`、`batch_valid_sample_ratio`。这样既能支持可读报告，也能支持之后的对比分析和参数历史可视化。

## 21. 结果目录与文件契约

结果目录必须采用“每次运行独立归档 + 根目录最新镜像”的双层策略。所有归档应放在 `results/runs/` 下，目录名由时间戳和模式组成。`sequential` 模式的归档目录应再按组细分，每组下保存各关节的 `capture` 和 `identification` 文件，summary 放在 `summary/` 子目录中。`collect` 模式则按批次直接保存 `hardware_capture_batch_xx.npz` 与 `hardware_identification_batch_xx.npz`，summary 直接位于本次 run 目录下。

除归档外，程序还必须把本次最新的 summary 镜像到根 `results/` 目录，包括 `hardware_identification_summary.npz`、`hardware_identification_summary.csv`、`hardware_identification_report.md` 和兼容 JSON。这样 `compensate` 模式才能直接读取根目录的最新 summary，而不必知道最近一次归档到底在哪个时间戳目录中。

每次运行都需要有一个 `run_manifest.json`。这个 manifest 至少要记录：run label、模式、开始时间、总轮次数、配置路径、归档路径、summary 文件路径、所有采样文件路径、所有辨识文件路径、补偿验证文件路径以及结束时间。manifest 的作用不是给人看，而是让程序自己能恢复和扫描历史运行。

## 22. 补偿模式契约

补偿模式复现时必须严格遵守“纯摩擦补偿”的原则。首先从 summary 中读取每个关节的 `coulomb`、`viscous` 和 `offset`。然后基于当前速度计算摩擦补偿力矩：

```text
tau_comp = coulomb * tanh(v / velocity_scale) + viscous * v + offset
```

计算出的补偿力矩要先经过关节力矩上限钳位，再按当前窗口模式决定是否执行接近边界的软衰减。最终下发的命令必须就是这个补偿力矩，不允许叠加轨迹前馈或反馈跟踪项。补偿模式的采样结果也要保存成独立文件，便于后续分析补偿效果。

## 23. 历史对比契约

`compare` 模式的核心是读取历史 summary，并生成一份面向人阅读的对比 Markdown 以及一份表格型 CSV。当前真实行为是只扫描 `collect` 归档，因此复现时也应只选取 `collect` 模式的历史结果。对比结果至少要包含每次运行的 run label、时间戳、批次数、归档路径，以及每个关节的 `coulomb`、`viscous`、`offset`、`validation_rmse`、`validation_r2`、`valid_sample_ratio` 和 `sample_count`。如果同时比较两次以上运行，还应提供“最新一次与前一次”的差分摘要。

## 24. 可视化契约

可视化不是主流程必要条件，但为了复现当前系统的完整行为，建议保留 `rerun` 仪表板。`rerun` 必须能够实时展示至少以下量：位置、速度、命令位置、命令速度、实测力矩、下发力矩、摩擦补偿力矩、残差力矩、旋转状态、范围比、边界余量、温度、UART 指标、有效样本比例和当前阶段。当前默认是 `focused` 模式，即主视图只聚焦当前正在辨识的关节；`full` 模式则显示全轴面板。无论采用哪种实现方式，都必须保证可视化初始化失败时主流程继续运行。

## 25. 复现顺序建议

如果要让另一个 AI 最快复现整套程序，最好的顺序不是按源文件逐个重写，而是按系统约束逐层落地。第一步实现配置加载和模式解析，先保证 `run.sh -> python -m` 这条启动链路打通。第二步实现串口上下行协议和全局状态缓存，先把“能收、能发、能记录”打通。第三步实现参考轨迹结构和控制循环，让 sequential 模式至少能跑通单关节采样。第四步实现离线前处理和摩擦拟合，让单轮 capture 能转换为 identification。第五步实现结果目录、manifest 和 summary 聚合，让程序从“能跑一次”升级为“能形成完整实验闭环”。第六步再补 collect、compensate、compare 和可视化。

如果资源有限，最小可用复现优先级应是：先实现 `sequential`，再实现 `compensate`，最后补 `collect` 和 `compare`。因为当前真实主线和根 summary 的消费方式都围绕 sequential 和 compensate 展开。

## 26. 一句话复现摘要

如果只保留一句最核心的复现说明，可以写成：

这套程序本质上是一个以真机串口反馈为输入、以 7 维力矩向量为下行控制量、以“平滑速度 + frictionless 刚体逆动力学残差”为辨识输入、以逐关节摩擦模型拟合为核心算法、以归档和 summary 为结果载体、并支持参数回放补偿验证的实验型流水线系统。只要把这个系统级契约完整复现出来，代码组织可以不同，函数名可以不同，但程序行为就会与当前仓库保持一致。
